**Prepared:** March 2026  
**Workspace:** Axiom repo (pushes to MarketX remote)  
**Author:** Anwesh Rath

---

## Executive Summary

MarketerX is a Revenue Share Operating System. Its single job is to manufacture booked calls at scale by running a structured belief-testing and email delivery machine. Every part of the system exists to support that outcome.

This plan maps the full product — what Tommy's documents define, what is already built, what is broken, what is missing, and the correct build sequence.

---

## What the Docs Define (Tommy's Source of Truth)

Tommy provided a complete, well-structured system with 4 document layers:

### 1. Rev Share Operating System
- **Phases 0–8**: Activation → ICP Build → Belief Construction → Delivery Infra → Controlled Ramp → Signal Engine → Flow Extension → Promotion Ladder → Scale
- All phases are gated. Nothing moves forward without exit criteria being met.
- Revenue model: Rev share only. No fixed-fee campaigns.
- Three engines: **Market Builder** (data), **Market Writer** (conversion), **Market Surface** (delivery)

### 2. MarketWriter System
- **Angle system**: 7 canonical angles (Problem Reframe, Hidden Constraint, False Solution, Economic Inefficiency, Market Shift, Opportunity Gap, Risk Exposure)
- **Belief as atomic unit**: One Brief = one ICP = two Beliefs (Champion vs Challenger)
- **Brief is immutable** after launch
- Every asset (email, page, post) references Brief_ID + Belief_ID + ICP_ID
- **Flow extension rules**: Flows extend in blocks (1–4, 5–8, 9–12, then Ongoing Reflection) only after signal thresholds are met
- **Website content**: Company website, email destination pages (Angle × ICP × Buyer Stage), SEO pages, social posts — all generated by MarketWriter

### 3. Measurement & Dashboard
- 5 dashboard layers: System Health → Deliverability → Belief Tracking → Campaign Performance → Sales/Revenue
- Belief Promotion Ladder: HYP → TEST → SW → IW → RW → GW
- Hybrid gating: 5 conditions must all pass before any promotion/allocation change
- Signal hierarchy: Primary (booked calls, positive replies), Secondary (click-to-reply), Observational (opens, raw clicks)

### 4. Implementation Guides
- Full data model: partner, offer, icp, brief, belief, flow, flow_step, asset, signal_event, belief_competition, belief_gate_snapshot, belief_promotion_log, meeting, opportunity, revenue_event, sending_identity
- Engineering checklist: all tables, constraints, state machines
- Implementation Mapping: fill this before any development begins

## The Correct Build Sequence

### Phase 1 — Foundation (RS:OS Schema + Signal Wiring)
**Goal:** Get the data model right so everything else can be built on it.

1. **Implement Tommy's full data model** from `MarketWriter Data Model.docx`:
   - Tables: partner, offer, icp, brief, belief, belief_competition, flow, flow_step, asset, signal_event, belief_gate_snapshot, belief_promotion_log, meeting, opportunity, revenue_event, sending_identity
   - All PKs as UUIDs
   - Brief immutability enforced (locked_at + DB policy)
   - Belief competition constraints (exactly 2 per brief, UNIQUE on belief_role)

2. **MailWizz Analytics Webhook** (`POST /api/imt/analytics/events`):
   - Receive CLICK, REPLY, BOOKED_CALL, BOUNCE, OPEN from MailWizz
   - Insert into signal_event table bound to Belief_ID + ICP_ID
   - This is the single most critical integration point

3. **Wire Learning Loop to signal_event**:
   - Update Learning Loop to read signal_event instead of (or in addition to) analytics_events
   - Ensure promoted/demoted variants actually flow back into flow generation

**Exit criteria:** Signal data flows from MailWizz → signal_event → Learning Loop → KB

---

### Phase 2 — MarketWriter Core
**Goal:** Get the belief-based email generation system working end to end.

1. **Brief generation service**:
   - Create Brief with all required fields from Tommy's schema
   - Enforce: one Brief = one ICP, exactly two Beliefs, immutable after launch

2. **Angle-bound flow generation**:
   - Each flow must reference Brief_ID + Belief_ID + Angle_Class
   - Generate Flow A (Champion) and Flow B (Challenger) bound to the same Brief
   - Initial block: emails 1–4 per flow, subject A/B per email, single CTA

3. **Flow Extension Logic**:
   - Evaluate gating conditions before extending to 5–8, then 9–12
   - Extension rules from `Flow Extension Rules (Canonical).docx`

4. **Belief Promotion Engine**:
   - State machine: HYP → TEST → SW → IW → RW → GW
   - 5-gate hybrid gating check before any promotion
   - Log all transitions to belief_promotion_log

**Exit criteria:** A Brief can be created, two flows generated, emails sent via MailWizz, signals returned, and belief promotion evaluated automatically.

---

### Phase 3 — Brain Integration + Self-Healing Loop
**Goal:** Close the learning loop so the Brain actually improves from outcomes.

1. **Brain ↔ Workflow integration**:
   - When intent = "write email", Brain looks up assigned engine and triggers it
   - Engine uses Brain's KB context for generation

2. **Self-healing loop**:
   - Learning Loop → updates Brain config (not just KB metadata)
   - When a belief reaches a certain promotion state → its angle class gets a weight boost in Brain config
   - RUST Brain API: feed signal_events → RUST Brain identifies KB gaps → pushes gap requests back

3. **Knowledge gap detection**:
   - In RAGOrchestrator: when relevance score < threshold or no docs found → return gap flag
   - Queue KB update request
   - Optional: RUST Brain researches and fills the gap

4. **Strict grounding mode**:
   - Brain mode flag: when enabled, reject or flag outputs that can't be attributed to retrieved docs
   - Return confidence score with every generation

**Exit criteria:** Brain improves belief generation based on what worked in previous campaigns. Knowledge gaps are detected and queued for filling.

---

### Phase 4 — Member Portal
**Goal:** Give clients a portal that shows them the gold.

1. **Client onboarding flow**:
   - Login → fill ICP questionnaire → assigned Brain is initialized
   - Or: team trains Brain on their behalf during onboarding call

2. **Member dashboard**:
   - Assigned flow (engine) view: which email flows are running, current status
   - Assigned Brain view: constitution, training status, chat interface
   - Data view: sends, replies, booked calls, revenue (the gold)
   - Chat with Brain: ask questions, get suggestions, train

3. **Tier-gated features**:
   - Basic: view-only insights, no chat, no manual training
   - Medium: insights + chat + manual training option
   - Enterprise: everything + write your own emails + custom flow builder

4. **Belief dashboard (simplified client view)**:
   - Show active beliefs, their status, which angle is winning
   - Don't expose full HYP/TEST/SW/IW/RW/GW ladder to basic clients
   - Show enterprise clients the full Belief Promotion Dashboard

**Exit criteria:** A client can log in, see what's running on their behalf, chat with their Brain, and see their campaign data.

---

### Phase 5 — Extended Content Generation (Website + SEO + Social)
**Goal:** MarketWriter generates all content types, not just emails.

1. **Email destination pages** (Angle × ICP × Buyer Stage):
   - Every email CTA links to a dedicated destination page
   - Page continues the same angle as the email
   - Structure: Hook → Recognition → Problem Reframe → Framework → Evidence → CTA

2. **Company website generation**:
   - MarketWriter generates all standard company pages
   - Single objective per page, single CTA

3. **SEO page generation**:
   - Worth Winning Keywords (authoritative long-form)
   - Long-Tail Keywords (broad coverage, one page per intent)
   - Internal linking structure: Long-Tail → Worth Winning

4. **Social post generation**:
   - Native posts (platform-only, no links)
   - Traffic posts (with CTA to destination page)
   - Each traffic post gets a unique destination page

**Exit criteria:** MarketWriter can generate a full content ecosystem: email flows + destination pages + website pages + SEO + social.

---

### Phase 6 — Scale + Multi-Tenancy + Reseller
**Goal:** System can handle multiple clients at scale with proper isolation.

1. **Per-client isolation hardening**:
   - Each client has isolated: ICP, Beliefs, Flows, Signals, Brain
   - No cross-client data leakage
   - Satellite/domain isolation enforced per partner

2. **Reseller / Mini Super Admin tier**:
   - Hierarchy: Platform Admin → Resellers → Clients → Users
   - Resellers can manage their own client accounts
   - Feature gating configurable per reseller

3. **Scale mechanics**:
   - Controlled ramp enforcement: one satellite at a time, 3k sends/day max
   - Volume doctrine: only scale after durability confirmed
   - Horizontal expansion: next satellite → next domain layer → scale tier

**Exit criteria:** Multiple clients can be managed simultaneously with full isolation, and resellers can be onboarded.

## RUST Brain Integration Strategy

Don't throw away the JS Brain. Use them together:

```
Client chat / email generation
         ↓
    JS Brain (RAGOrchestrator + Agents)
    - Fast response
    - KB retrieval
    - Multi-agent routing
         ↓
  Generated emails sent via MailWizz
         ↓
    Signal events (replies, bookings, revenue)
         ↓
    RUST Brain (self-improvement engine)
    - Analyzes signal patterns
    - Detects KB gaps
    - Runs research in sleep mode
    - Pushes gap fills + weight updates
         ↓
    JS Brain KB updated
    → Better generation next cycle
```

RUST Brain = overnight optimization engine  
JS Brain = real-time generation engine  
They improve each other via signal feedback loop.
