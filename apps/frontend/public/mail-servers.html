Section 01

Executive Summary

After analyzing Gmail's reputation tracking mechanics and evaluating multiple architectural approaches, we recommend a 5-TLD Shield Model — five completely separate domains, each hosted on a single dedicated SMTP server with 10 mailbox identities, producing a total capacity of 150,000 emails per day.

Core Finding

Gmail tracks reputation at the organizational domain level, not the IP or subdomain level. This means adding more IPs or subdomains under the same domain provides negligible additional protection. The real isolation boundary is the top-level domain (TLD) itself. Five separate TLDs give you five independent reputation silos — if one burns, the other four are completely untouched.

Section 02

Key Discovery: How Gmail Actually Tracks Reputation

Gmail's spam filtering evaluates sending reputation across multiple signals. Understanding which signal dominates is critical to designing the right architecture.

The Three Reputation Layers

LayerWhat Tracks ItIsolation LevelSignificance
Domain Reputation Gmail Postmaster Tools, internal signals Per organizational domain (e.g., brand.com) PRIMARY — This is what matters most
IP Reputation Spamhaus, Barracuda, RBLs, Gmail Per IPv4 address Secondary — affects RBL-based filtering
Subdomain Reputation Rolls up to parent domain No independent isolation None — provides zero isolation

Critical Implication

If sarah@brand.com sends spam from IP-A, and mike@brand.com sends clean email from IP-B — both share brand.com's domain reputation. Gmail's primary signal is the domain, not the IP. Mike's clean email will suffer because Sarah's behavior damaged the shared domain reputation. This is true regardless of whether they use the same IP, different IPs, or different subdomains.

What This Means for Architecture Design

The only way to achieve true reputation isolation is to use completely different root domains (different TLDs). Two emails from brand-alpha.com and brand-beta.io have fully independent reputations in Gmail's system. One can burn without affecting the other.

The Protection Hierarchy

Strong isolation: 5 different TLDs — completely independent reputations. If one burns, four survive.
Moderate benefit: Multiple IPs per TLD — limits blast radius on IP-based blacklists (Spamhaus, etc.). Does not protect against domain reputation damage.
No benefit: Subdomains — Google tracks the organizational (parent) domain. Subdomains add management complexity with zero isolation gain.

Section 03

Why Subdomains Do Not Provide Isolation

A common misconception in email infrastructure is that using subdomains (e.g., sub1.brand.com, sub2.brand.com) creates independent reputation silos. This is incorrect.

Side-by-Side Comparison

ScenarioWhat You ExpectWhat Gmail Actually Does
sub3.brand.com gets flagged for spam Only sub3 is affected; sub1, sub2 continue normally brand.com domain reputation drops. ALL subdomains are affected.
10 subdomains on 10 different IPs 10 independent sending identities 1 domain (brand.com) with 10 IPs. Google sees one entity.
Move bad subdomain to a new IP Fresh start for that subdomain Domain reputation persists. New IP doesn't fix domain-level damage.

Root Domain Identities vs. Subdomains — Identical Outcome

Attributesarah@brand.com (root)sarah@sub1.brand.com (subdomain)
Gmail sees domain asbrand.combrand.com (same)
Domain reputationbrand.combrand.com (same)

DNS records needed1 SPF, N DKIM selectors, 1 DMARCN SPFs, N DKIMs, N DMARCs (10x more)

Conclusion

Since the outcome is identical, there is no reason to use subdomains. Root domain identities (sarah@brand.com, mike@brand.com) are simpler to configure, simpler to manage, and carry zero additional risk compared to subdomains. We recommend root domain identities exclusively.

Section 04

Three Models Evaluated

We evaluated three architectural approaches against the hard requirements: 5 TLDs per client, up to 150K emails/day, and maximum isolation.

Model A: 50 Servers — Maximum IP Isolation

Structure: 5 TLDs × 10 servers each. One mailbox identity per server, one IP per server, 3K/day per server.
Cost: 50 × $5/mo = $250/mo (cheap VPS) or 50 × $50/mo = $2,500/mo (optimized SMTP)
Pros: Maximum IP-level isolation. Each identity has its own IP.
Cons: Domain reputation risk is identical to Model C. 50 servers to manage. 10x–50x the cost for marginal IP-blacklist protection only.

Model B: 50 Subdomain Servers

Structure: 5 TLDs × 10 subdomains, each on its own VPS.
Cost: 50 × $5–$50/mo = $250–$2,500/mo
Pros: None over Model A.
Cons: All cons of Model A, plus: subdomains do not provide domain reputation isolation. 10x DNS management overhead. Rejected — strictly worse than Model A.

Model C: 5 Servers — Maximum Efficiency (Recommended)

Structure: 5 TLDs × 1 server each. 10 mailbox identities per server, each identity capped at 3K/day. Each server handles 30K/day (its rated capacity).
Cost: 5 × $50/mo = $250/mo
Pros: Same domain-level protection as Model A. 10x cheaper. 10x easier to manage. Pre-configured SMTP boxes from trusted vendor. Each box runs at rated capacity — no waste.
Cons: 10 identities share 1 IP per domain. If the IP gets RBL-blacklisted, all 10 identities on that domain pause until replacement.

Key Design Principle

The RackNerd command center (IPs .66 through .78) never sends any email. RackNerd hosts MailWizz (the orchestrator) and Refinery Nexus (the data warehouse). All actual email delivery goes through satellite servers with their own IPs. RackNerd IPs are not present in any sending domain's SPF record. This completely isolates the command center from any sending reputation risk.

Section 06

DNS Configuration Per Domain

Each of the 5 sending domains requires the following DNS records. All records point to the satellite server's IP — never to the RackNerd command center.

SPF Record

; Authorizes only the satellite server's IP to send for this domain
Type:  TXT
Name:  @
Value: v=spf1 ip4:{satellite_ip} ~all

DKIM Records (10 selectors per domain)

; Each mailbox identity signs with its own DKIM key
; Generated on the satellite server via opendkim-genkey
Type:  TXT
Name:  s1._domainkey     → public key for identity 1 (sarah@)
Name:  s2._domainkey     → public key for identity 2 (mike@)
Name:  s3._domainkey     → public key for identity 3 (emily@)
... through s10._domainkey

DMARC Record

Type:  TXT
Name:  _dmarc
Value: v=DMARC1; p=quarantine; pct=100; aspf=r; adkim=r;

MX Record

Type:  MX
Name:  @
Priority: 10
Value: mail.{domain}

PTR / Reverse DNS

; Set at the satellite server provider's panel — NOT in Cloudflare/Namecheap
{satellite_ip}  →  mail.{domain}

Important

PTR records must be configured at the satellite SMTP provider's control panel. This is the #1 missed step that kills deliverability. Each satellite IP must resolve backwards to mail.{domain}.

Self-Healing Behavior

If any satellite exceeds a bounce rate of 2% or a spam complaint rate of 0.1%, the SendPacingAgent automatically pauses that satellite. Traffic is redistributed to healthy satellites. The burned satellite can be replaced with a fresh $50 server and a new domain, with a 14-day warmup before returning to full capacity.

Risk Matrix & Burn Scenarios

EventImpactSurviving CapacityRecovery
1 satellite IP gets RBL-blacklisted That TLD's sending pauses. Other 4 TLDs unaffected. 120,000/day (80%) Replace server ($50), request delisting, or switch to new IP. 24–72 hours.
1 TLD domain reputation burns at Gmail All 10 identities on that domain go to spam. Other 4 TLDs unaffected. 120,000/day (80%) Retire domain. Purchase new TLD. Fresh server + 14-day warmup. Cost: $50/mo + $15 domain.
Satellite server hardware failure 1 TLD offline temporarily. Zero data loss (stateless SMTP relay). 120,000/day (80%) Spin new server from vendor. Reconfigure DNS. 1–4 hours.
RackNerd command center goes down MailWizz + Refinery offline. New campaigns can't dispatch. Already-queued emails on satellites still deliver. In-flight emails continue. New sends paused. RackNerd SLA 99.9%. If prolonged, MailWizz can be moved to any server — it's the orchestrator, not the sender.
Worst case: all 5 TLDs burn simultaneously Extremely unlikely with 3K/day caps and auto-pause at 2% bounce. Would require systematic list quality failure across all domains. 0/day 5 new domains + 5 new servers. $250/mo + $75 domains. 14-day warmup. Total restart cost: ~$325.

Why Total Burnout Is Extremely Unlikely

The SendPacingAgent enforces a hard cap of 3,000 emails per identity per day. Auto-pause triggers at >2% bounce rate or >0.1% complaint rate. At these conservative thresholds and volumes, getting a domain blacklisted requires sustained bad list quality that the verification engine and Coach Agent are specifically designed to prevent. The self-healing loop catches problems before they become catastrophic.
