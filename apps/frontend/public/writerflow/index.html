PHASE: DESIGN TIME

The Blueprint Layer

These entities are created by Superadmin. They define WHAT can be deployed. Nothing here touches a real org until deployment.

🧠
Brain Template
Master AI config blueprint — providers, RAG settings, memory limits, prompt layers. The "DNA" of an org's intelligence.
Table: brain_templates
Key columns: name, version, config (JSON), pricing_tier, foundation_layer_id, persona_layer_id, guardrails_layer_id
Contains: Provider config, RAG settings (topK, confidence, weights), memory config, rate limits, inline agent stubs
Role: Cookie cutter — stamps out brain_agents at deploy time. Template is COPIED, never referenced at runtime.
Prompt source: References prompt_layers table for foundation/persona/guardrails text
🤖
Agent Template
Reusable agent archetype — personality, prompts, model prefs, tools, skills. Interactive agents that chat, write, and communicate.
Table: agent_templates
Key columns: slug, name, system_prompt, persona_prompt, instruction_prompt, guardrails_prompt, preferred_model, tools_enabled, skills, category, product_target
Categories: writer, research, learning, builder, general
Products: market_writer, market_builder, market_coach, all
Role: Another cookie cutter — gets cloned into org_agents at engine deployment. Can be linked to brain templates via brain_agent_assignments.
⚡
Workflow Template
Visual orchestration graph — nodes and edges (ReactFlow compatible). Defines the execution sequence.
Table: workflow_templates
Key columns: name, nodes (JSON[]), edges (JSON[]), status (draft/active/disabled)
Example: "Email Nurture Flow" — nodes for ICP analysis → angle selection → email drafting → quality check
Role: Frozen into engine snapshot at deploy. Workflow graph becomes the execution plan for the worker.
📝
Prompt Block
Modular, reusable prompt components. Categories: foundation, persona, instruction, guardrails, domain, task, output, analysis, optimization, compliance.
Table: prompt_blocks
Key columns: slug, name, category, content, tags, version, is_system
Assignment: prompt_assignments links blocks to brain_agents, agent_templates, or org_agents
Role: Composable prompt library. Blocks are APPENDED to base prompts at runtime — never replace. Think "plugins" for prompts.
Currently: 15 seed prompts across 10 categories. Phase 3 of the plan wires these into runtime.
🎯
Mastery Agent
Headless agent — intelligent but no UI. Runs silently in the pipeline making decisions (timing, angles, contact priority).
Table: mastery_agent_configs
Key columns: agent_key, decision_type, decision_outputs, scoring_rules, keyword_rules, field_rules, pipeline_stage, pipeline_order
Pipeline stages: pre_send, post_reply, pre_extension, periodic, on_demand
Categories: contact, timing, angle, pacing, reply, buying_role, buyer_stage, uncertainty, sequence, custom
Role: The invisible hand — shapes every outreach decision without user interaction. Rule-based scoring, not LLM calls (fast).
📦
Engine Bundle
Deployable package: Brain Template + Workflow Template + Agent configs + LLM defaults + BYOK mode. This is WHAT gets deployed.
Table: engine_bundles
Key columns: name, slug, brain_template_id, workflow_template_id, agents_config (JSON[]), default_llm (JSON), tier, email_provider_id, default_api_key_mode
Agents config: Array of agent definitions — each with role, name, prompts, LLM overrides, tools, RAG settings
Role: The master package. Everything needed to give an org a fully functional AI engine. Think "app template" for orgs.

The Stamping Process

What Gets Created on Deploy

🧠 brain_agents row(s)
One row per agent role. Prompts, RAG config, model prefs, tools — all COPIED from the template. The template_id field is kept for audit only.

Key: foundation_prompt, persona_prompt, domain_prompt, guardrails_prompt — these are the runtime prompts.
🤖 org_agents row(s)
One row per agent role. Full agent config — prompts, LLM settings, tools, skills, KB access. Links to both the brain_agent and the source agent_template.

Key: engine_instance_id, brain_agent_id — these link everything together.
⚡ engine_instances row
The deployed engine. Contains a frozen snapshot of the entire bundle at deploy time — workflow nodes/edges, agent configs, LLM defaults, brain reference.

Key: config.flowConfig (workflow), snapshot (full audit), api_key (unique per deploy).
🔑 API Key
Unique axm_live_* key generated per deployment. SHA-256 hashed in DB. Shown once to admin — must be copied.

Modes: platform (Axiom's keys) or byok (client's own keys from ai_providers).
⚠️ Critical: Templates Die After Deploy
The deploy code literally says: // Prompt layers — copied as text (NOT referenced)

Once deployed, the engine instance and brain_agents are the ONLY runtime truth. If you update a brain template, existing deployments are NOT affected. You must re-deploy to push changes. This is BY DESIGN — it prevents live systems from breaking when templates are edited.
PHASE: RUNTIME EXECUTION

The Execution Pipeline

When a user triggers a Writer run (or any engine execution), this is the exact sequence that happens.

1. Feature Gate Check
requireFeature(req, 'can_write_emails')
Verifies the user's tier allows Writer access. Returns orgId and userId.
2. Engine Resolution PHASE 1 — NEW
requireActiveEngineRuntime(orgId, userId)
Finds the deployed engine_instance for this org. Loads workflow snapshot, brain agent, org agents, LLM config, BYOK mode. Replaces the old name-based workflow lookup and auto-create shortcut.
3. Context Assembly
brainKBService.buildWriterContext(brainAgentId, orgId, ...)
Loads: KB content → ICP details → Beliefs → Offers. Plus user's custom instructions, angle, goal.
4. Prompt Stack Assembly PHASE 3 — NEW
assemblePromptStack(brainAgentId, inlinePrompts)
Base prompts from brain_agent + modular prompt_blocks from Prompt Studio. Blocks are APPENDED by category. Foundation + Persona + Domain + Guardrails + Instructions.
5. Mastery Pre-Send PHASE 6 — NEW
runMasteryPipeline('pre_send', orgId, input)
Headless mastery agents score the input: optimal angle, timing, contact priority. Enriches the input before workflow execution. Rule-based, sub-100ms.
6. Queue to Redis
Job queued to engine-execution with full payload: engine snapshot, prompt stack, KB context, ICP, beliefs, offer, mastery decisions, orgId (for BYOK), LLM config.
7. Worker Execution
engine-execution-worker on your server picks up the job. Executes workflow nodes using aiService.callWithOrgContext(orgId, ...) — resolves BYOK chain: org keys → platform keys. Real AI calls happen here.
8. Post-Reply Validation PHASE 7 — NEW
Mastery agents run post_reply checks: quality scoring, compliance, tone analysis. If quality is below threshold → self-healing retry (max 2 attempts). Gaps logged to knowledge_gaps for learning loop.
9. Output Stored
Final output written to engine_run_logs (status: completed, tokens, cost, duration). Also updates the runs table for the user's Writer Studio view. Progress published via Redis Pub/Sub.

The Prompt Assembly Chain

How the final system prompt is built — layer by layer. This is what the LLM actually sees.

Layer 1 — Foundation Prompt
Source: brain_agents.foundation_prompt
"You are an expert email copywriter for [brand]. Your role is to..."
+ any foundation prompt_blocks appended
Layer 2 — Persona Prompt
Source: brain_agents.persona_prompt
"Write like a confident, direct SaaS founder. No jargon. No fluff..."
+ any persona prompt_blocks appended
Layer 3 — Domain Prompt
Source: brain_agents.domain_prompt
Industry-specific context: "In the B2B SaaS space, buyers expect..."
+ any domain prompt_blocks appended
Layer 4 — Guardrails Prompt
Source: brain_agents.guardrails_prompt
"NEVER make false claims. NEVER promise results. NEVER defame competitors..."
+ any guardrails/compliance prompt_blocks appended
Layer 5 — Instruction Blocks NEW via Prompt Studio
Source: prompt_blocks (instruction, task, output, analysis, optimization categories)
Modular instructions: "Structure emails as PAS framework", "Include CTA in every email"
Layer 6 — Runtime Context (injected per run)
KB Content: Brand voice, examples, reference material from kb_sections
ICP: Target customer pain points, demographics, criteria
Beliefs: Brand positioning / belief frameworks from beliefs
Offer: Value prop, primary promise from offer
User Instructions: Free-text from the writer/new form
Angle + Goal: Problem reframe, social proof, etc.
✅ Final Assembled Prompt → LLM
All layers concatenated. Sent to the AI provider via the BYOK chain. The LLM sees ONE unified system prompt, not separate layers.
API KEY RESOLUTION

BYOK vs Platform-Managed

The AI provider chain — how API keys are resolved for every LLM call.

🏢 Platform-Managed
Engine deployed with api_key_mode = 'platform'

Key source: ai_providers WHERE org_id IS NULL
Who pays: Client pays Axiom → Axiom pays provider
Control: Axiom manages keys, rate limits, failover
Best for: SMBs who don't want to deal with API keys
🔑 BYOK (Bring Your Own Key)
Engine deployed with api_key_mode = 'byok'

Key source: ai_providers WHERE org_id = orgId
Who pays: Client pays provider directly
Control: Client manages their own keys, models, limits
Best for: Enterprise clients with existing AI contracts

HEADLESS INTELLIGENCE

Mastery Agents — The Invisible Hand

They never talk. They never show UI. But they shape every decision behind the scenes.

🎯 Pre-Send Pipeline
Runs BEFORE the workflow executes. Makes decisions about:

Angle Selection: Which messaging angle is optimal for this ICP?
Timing: Best time to send based on engagement data
Contact Priority: Which contacts should go first?
Sequence Pacing: How many days between emails?

Rule-based scoring — sub-100ms execution. Not LLM calls.
🔍 Post-Reply Pipeline
Runs AFTER the workflow completes. Quality gates:

Quality Scoring: Does the output meet minimum standards?
Compliance Check: Any forbidden patterns or false claims?
Tone Analysis: Matches the brand voice from persona?
Self-Healing: If quality < threshold → auto-retry (max 2)

Gaps logged to knowledge_gaps for the learning loop.

IMPLEMENTATION PLAN

7-Phase Unification Plan

The roadmap to connect everything properly. Phase 1 must go first. No shortcuts.

P1 Engine Instance Resolver
New service: EngineInstanceResolver.ts. Single source of truth for resolving the active deployed engine for any org. Every execution path goes through this. Returns full runtime: workflow, brain agent, org agents, LLM config, BYOK mode.
P2 Rewire Writer Studio
Kill the shortcut. Writer goes through the deployed engine. No more hardcoded name lookups, no more phantom engine creation. If no engine is deployed → loud error. File: /api/writer/execute/route.ts
P3 Wire Prompt Blocks into Runtime
Make prompt_blocks + prompt_assignments affect agent behavior. Blocks are APPENDED to base brain prompts by category. Prompt Studio becomes the modular layer. Changes take effect without redeploying.
P4 Wire BYOK into Engine Execution
Pipe orgId through to the worker so callWithOrgContext() resolves org BYOK → platform keys. The chain exists — just needs the org context threaded through.
P5 Kill Hardcoded Constants
Move angle classes, flow goals, email defaults into engine_bundles.config. Writer /new reads from engine config via API instead of hardcoded arrays.
P6 Mastery Agent Pipeline
Wire headless mastery agents into pre_send/post_reply stages. Before workflow: optimize angle/timing/priority. After workflow: quality check, compliance, tone analysis.
P7 Validation & Guardrails
Self-healing quality loop: if output quality < threshold → regenerate (max 2 attempts). Gaps logged to knowledge_gaps for learning loop. Strict grounding enforcement.
📌 Dependency Order
P1 → P2 + P3 (parallel) → P4 + P5 (parallel) → P6 → P7
Phase 1 must be done first — everything depends on the Engine Instance Resolver.
No schema changes needed. All existing tables stay. Just rewiring the connections.
